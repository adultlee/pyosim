# Known Issues

## [UI] 3회 쌍둥이 건수 과대 계상 — 투표소 단위 원본

**발견일:** 2026-06-16  
**영향 범위:** UI의 회차별 건수 표시, 3회(2002) 수치 신뢰도

### 현상

`twin_votes_3.json`의 쌍둥이 그룹 수가 18,426개로 다른 회차(4회 1,462개, 5회 894개)에 비해 압도적으로 많다.

### 원인

3회(2002) 원본 XLS가 읍면동 단위가 아니라 **투표소 단위**(예: `신안동제1투`, `화서1동제2투`)로 집계돼 있다. 같은 읍면동 안의 투표소끼리도 득표가 같으면 쌍둥이로 잡히기 때문에 건수가 과대 계상된다.

5회(2010) 894건은 이상한 게 아니라 읍면동 단위로 정상 집계된 결과다.

### 현재 상태

ETL 파서(`etl/local/parse_3rd.py`)는 원본 구조 그대로 파싱하고 있어 투표소 행이 읍면동처럼 CSV에 들어간다. 데이터 자체는 원본 충실이나 분석 목적(읍면동 단위 비교)에는 맞지 않는다.

### 해결 방향

- **ETL 수정안:** `parse_3rd.py`에서 투표소 행을 읍면동 단위로 합산 후 emit. 단, 원본 구조 변경이라 검증 로직도 함께 수정 필요.
- **UI 임시 대응안 (우선 적용):** 3회 회차 버튼 옆에 안내 문구 표시. 예: "3회는 투표소 단위 집계로 건수가 실제보다 많을 수 있습니다."

### 결정

UI에서 먼저 안내 문구로 대응한다. ETL 수정은 별도 작업으로 분리.

## [분석] 쌍둥이 집계 방식 재설계 필요

**발견일:** 2026-06-16

### 문제 1 — 2-조합 방식으로 중복 과다

현재 `analyze_twin_votes.py`가 투표구당 후보 전체의 2-조합을 버킷 키로 사용한다.  
A·B·C 세 읍면동이 같은 패턴이면 A↔B, A↔C, B↔C 세 그룹으로 분리되는데, 이는 "하나의 사건"이다.

**결정:** 상위 N위 쌍 기준 클러스터로 변경.  
같은 `(회차, 선거종류, 비교단위, level, 순위쌍)` 키를 공유하는 읍면동 전체를 하나의 클러스터로 묶는다.  
필터는 `1·2위`, `2·3위`, `3·4위` 등 순위 쌍별로 제공하며 중복 허용.

### 문제 2 — 그룹 키에 시도가 고정돼 시도 경계를 넘는 선거를 못 잡음

현재 키: `(회차, 선거종류, 시도, level)`  
9회 광주+전남 통합 시도지사 선거처럼 시도가 다른 투표구끼리 비교돼야 하는 경우를 놓친다.  
(실제로는 `광주 송정1동 ↔ 전남 고흥 금산면` 클러스터가 잡히긴 했으나, 우연히 잡힌 것)

올바른 그룹 키는 선거종류마다 다름:

| 선거종류 | 비교 단위 |
|---|---|
| 시도지사 | 시도 (통합선거는 별도 매핑 필요) |
| 구시군장 | 구시군 |
| 시도의원 | 선거구 |
| 구시군의원 | 선거구 |
| 광역비례 | 시도 |
| 기초비례 | 구시군 |
| 교육감 | 시도 (교육청 단위) |
| 교육의원 | 선거구 |

**결정:** `analyze_twin_votes.py` 버킷 키 로직을 선거종류별 비교단위 매핑으로 교체.  
광주+전남 통합 등 예외는 별도 매핑 테이블로 관리.

### 문제 3 — 동점 시 순위 정렬 기준 불안정

득표수가 같은 후보가 여러 명일 때 이름 알파벳 정렬을 쓰면 선관위 기호순과 달라져 동점 케이스를 놓침.  
(강진 병영면 이정현·이종욱 동점 케이스 미검출)

**결정:** CSV 행 순서(원본 기호순 반영)를 우선 정렬 기준으로 사용.
