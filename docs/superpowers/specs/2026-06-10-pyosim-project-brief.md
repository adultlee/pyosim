# 표심 (pyosim) — 프로젝트 브리프

## 한 줄 정의
중앙선관위에 흩어져 있어 찾기 어려운 한국 선거 결과 데이터를, **검색·비교·다운로드**가 쉬운 단일 아카이브 사이트로 재가공한다.

## 배경 / 문제
선거 결과는 선관위(공공데이터포털·국가선거정보 개방포털)에 공개돼 있지만, 회차·선거종류별로 파일이 흩어져 있고 원본 포맷이 거칠어 일반 사용자가 원하는 결과를 찾고 비교하기 어렵다. 데이터 자체의 가치보다 **접근성·탐색성·재사용성**에 가치를 둔다.

## 핵심 가치 (우선순위)
1. **빠른 검색/조회** — 특정 선거·지역 결과를 즉시 찾기
2. **비교 분석** — 회차별·지역별 비교
3. **원본 데이터 다운로드** — 가공 전 원본을 받을 수 있게 제공

## 데이터
- **출처**
  - 공공데이터포털(data.go.kr): 코드정보 API(선거ID·선거종류코드 → 다른 호출의 전제), 투·개표 정보 조회 API, 개표결과 파일데이터(CSV/XLSX)
  - 국가선거정보 개방포털(data.nec.go.kr): 파일데이터·OpenAPI·LOD (보조)
- **범위(목표)**: 대통령선거 13~21대 / 국회의원선거 14~22대 / 전국동시지방선거 3~8회 / 재보궐 2010년 이후
- **그래뉼래리티**: 읍면동·투표구까지 풀 데이터 적재
- **갱신 특성**: 실시간 아님. 선거 종료 후 통상 2개월 이내 검증 후 공개 → 정적/배치 처리에 적합 (실시간 크롤링 비권장)

## ETL 주의점 (원본 XLSX 실측으로 확인된 복병 — 반드시 반영)
선관위 개표결과 XLSX는 그대로 DB에 못 넣는다. 단일 파서가 아래를 모두 흡수해야 한다.

1. **헤더가 2~3줄 병합 셀**: 0행=대분류(`후보자별 득표수`가 여러 칸 병합), 1행=`정당1`/`후보1` 같은 placeholder, **실제 후보·정당명은 2행(0-index)**. 데이터는 3행부터.
2. **파일마다 위치 컬럼 구조가 다름**: `선거구명` 유무·순서가 제각각이고, 시도지사·교육감 파일은 첫 컬럼 라벨이 `선거구명`이지만 값은 실제 시도명. → **첫 컬럼은 라벨 무시하고 항상 sido로 취급**, 나머지는 헤더 문자열로 role 매핑.
3. **후보 셀이 `정당명\n이름` 한 칸**: 지역구는 `\n`으로 split, 비례는 정당명만(이름 없음). 후보 칸 수 가변(2~12+), 빈 칸은 `'\n'`/`'0'` padding → 실제 후보만 필터.
4. **행 계층이 `구분`/`읍면동명`에 숨어있음**: `합계`(구시군), `거소투표`/`관외사전투표`(구시군 특수), `소계`(읍면동), `관내사전투표`(읍면동 특수), 개별 투표구. **집계행을 안 거르면 득표가 약 3배 뻥튀기됨.** → 모든 행을 저장하되 `level` 컬럼으로 태깅하고, 집계 시 한 레벨만 필터.
5. **숫자가 `"129,816"` 콤마 문자열**, 시트명에 중점(`구·시·군의회의원선거`) 포함.
6. **선거 식별자(sgId·선거일·회차)가 파일 안에 없음** → 파일명/수동 태깅 또는 코드정보 API와 매핑 필요. 받은 파일에 출처가 다른 선거(예: 단일 선거구만 든 국회의원 파일)가 섞일 수 있으니 적재 전 출처 검증.

## 비교 분석 제약 (중요 — 약속 못 지키면 신뢰 문제)
- 투표구는 선거마다 재편되어 코드·명칭 불일치 → **회차 간 투표구 단위 비교는 불가**.
- 읍면동도 행정구역 통폐합(예: 청주+청원, 세종 신설)으로 시계열 깨짐.
- **회차 간 안정적 비교 최소 단위 = 시군구**.
- → 검색/다운로드용 원본은 투표구까지 풀 적재, **비교 분석용은 시군구 집계 + 행정구역 변경 매핑 테이블**을 별도로 둔다.

## 기술 스택 / 아키텍처
- 프론트: Next.js (Pages Router) + TypeScript + Tailwind CSS
- DB: Supabase (Postgres). 풀 적재 시 무료 티어 500MB 빠듯 → Pro 검토, 또는 투표구 원본은 정적 파일로 빼는 하이브리드
- 데이터 흐름: API/파일 → 정규화(ETL) → Supabase 적재 → 프론트는 ISR/빌드타임 생성(데이터가 정적이라 적합)
- 다운로드: 온디맨드 쿼리→CSV 변환 대신, 회차/선거별 CSV·JSON을 미리 생성해 R2/Storage에 정적 배치 후 링크 제공
- 모노레포(안):
  ```
  pyosim/
  ├─ apps/web/        # Next.js 프론트
  ├─ packages/etl/    # xlsx → tidy → Supabase 파서
  ├─ packages/db/     # Supabase 스키마·마이그레이션
  └─ data/raw/        # 원본 xlsx (gitignore 또는 LFS)
  ```

## 비용 목표
- 가벼운 freemium/취미·포트폴리오 성격, 목표 ~₩100,000/월 수준 유지.
- 정적 생성 + 정적 파일 다운로드로 호스팅·DB 부하 최소화가 핵심 전략.
