# 검증된 지방선거 ETL 파이프라인 — 설계

## 한 줄 정의
원본 XLSX → CSV 변환이 **자동 검증을 통과한** 지방선거(3~8회) 데이터 파이프라인. 검증을 통과하지 못하면 파이프라인이 실패하고, 어느 파일·어느 시도가 왜 틀렸는지 리포트한다.

## 배경 / 문제

현재 `process_local_elections.py`는 9개 샘플 XLSX로만 검증됐고, 전 회차를 돌렸더니 데이터가 깨졌다:

- **8회 시도지사: 전국 17개 시도가 전부 서울 후보(오세훈·송영길·권수정·신지혜·김광종)로 채워짐.** 통합 파일(`시도지사선거.xlsx`)에서 각 시도 블록 첫 행에 그 시도용 후보 헤더가 다시 나오는데, 파서가 맨 위 서울 헤더만 한 번 읽어 전국에 적용했다.
- **양산면 등에 같은 후보가 중복 행으로 여러 번** 적재됨 — 집계행/투표구 처리 꼬임.
- 이 오류가 검증 없이 CSV → 쌍둥이 분석(`twin_votes.json`) → UI까지 그대로 흘러갔다.

근본 원인: **ETL이 원본을 제대로 옮겼는지 검증하는 장치가 없다.**

## 검증의 의미 (중요)

검증은 **"내 파서가 원본을 손실·왜곡 없이 그대로 옮겼는가"**를 보는 것이지, 원본 데이터가 틀렸다고 말하는 게 아니다. 원본(선관위 공식 데이터)은 항상 옳다. 검증이 빨간불이면 **100% 파서 잘못**이며, 고쳐서 초록불 될 때까지 간다. 초록불 = 원본을 그대로 옮겼다는 보증.

검증 항목은 전부 "제대로 베꼈나" 검사다. 원본을 의심하는 항목은 하나도 없다.

## 목표 상태

> 원본 XLSX → CSV 변환이 자동 검증을 통과한 지방선거(3~8회) 데이터 파이프라인.
> 검증을 통과하지 못하면 파이프라인이 실패(exit≠0)하고, 위반 항목을 구조화해 리포트한다.

작업 범위: **지방선거 3~8회 먼저.** 패턴이 잡히면 대통령/국회의원/재보궐은 같은 방식으로 확장.

## 아키텍처

```
원본 XLSX                  parse                  validate                 적재
data_raw/지방선거/  ──▶  파일별 파서  ──▶  tidy rows  ──▶  검증 게이트  ──▶  지방선거.csv
                         (회차×선거종류)                   (통과해야 통과)
```

핵심 원칙: **검증을 통과하지 못하면 CSV를 쓰지 않는다.**

### 구성 요소

**(A) 파서 모듈** — 회차별로 원본 구조가 다르므로 분리. 각각 Opus 서브에이전트가 원본을 직접 까보고 설계/구현.
- `parse_3rd`, `parse_4th`, `parse_5th`, `parse_6th`, `parse_7th`, `parse_8th`
- 공통 출력 tidy row 컬럼:
  `선거_회차, 선거일, 선거종류, 시도, 구시군, 읍면동, 선거구명, 선거인수, 투표수, 후보자, 정당, 득표수, 무효투표수, 기권수, level`

**(B) 검증 모듈** `validate_local.py` — 이번 작업의 심장. tidy rows를 받아 검증.

내부 정합성 (필수 게이트, 위반 시 실패):
1. **시도별 후보 일관성** — 한 시도지사 선거에서 (ㄱ) 한 시도 내 모든 읍면동의 후보 집합이 동일하고, (ㄴ) 시도마다 후보 집합이 서로 달라야 한다. 같으면 = 파서가 헤더를 잘못 복사함. (8회 버그 직격)
2. **합계 정합성** — level별로 분리 집계했을 때 투표구 합 == 원본에 적힌 소계·합계. 안 맞으면 = 파서가 행을 빠뜨리거나 중복함.
3. **중복 행 없음** — `(선거_회차, 선거종류, 시도, 구시군, 읍면동, level, 후보자)` 유일. 있으면 = 같은 행을 두 번 읽음.
4. **범위 정합성** — 후보 득표 ≤ 투표수 ≤ 선거인수. 위반 = 컬럼을 밀려 읽음.

외부 대조 (가능한 회차만):
- 시도지사 선관위 공식 시도별 1·2위 득표와 CSV 합계 대조. 위키백과/나무위키 결과표에서 검증용 핵심 표본만 확보(전수 불필요). 17개 시도 1위 득표가 맞으면 시도 매핑·헤더 갱신이 올바르다는 강력한 증거.

**(C) 오케스트레이터** `build_local.py` — 파서 호출 → 검증 → 통과 시에만 CSV 작성, 리포트 출력.

### 검증 실패 리포트 형식

```
[FAIL] 8회 시도지사: 시도별 후보 불일치
  - 충청북도 후보집합 == 서울특별시 후보집합 {송영길,오세훈,...} → 시도 헤더 갱신 누락 의심
[FAIL] 8회 시도지사 영동군 양산면: 중복 행 3건 (송영길/당일투표)
```

## 회차별 원본 구조 (파서가 흡수해야 할 것)

- **3회**: 선거종류별 폴더 → 압축 풀린 개별 XLS(시도/구시군별 파일). A형(시도지사/광역비례, col0=위원회명) / B형(구시군장/시도의원, col0=투표구명) 두 구조.
- **4회**: 번호+선거종류 폴더 (`1_시도지사` 등).
- **5회**: 번호+선거종류 폴더 (`01_시도지사` 등).
- **6회**: **시도별로 분리된** 폴더 (강원/경기/...).
- **7회**: 선거종류별 통합 XLSX 하나 (`7지선-01-(시도지사)`). 시도 전환 지점에서 후보 헤더 갱신 필요.
- **8회**: 선거종류별 통합 XLSX 하나 (`시도지사선거.xlsx`). 시도 블록 첫 행(읍면동명 빈칸 + 후보셀에 `정당\n이름`)이 그 시도 헤더. **현재 깨진 부분.**

선거종류: 시도지사, 광역비례, 구시군장, 시도의원, 구시군의원, 기초비례, 교육감, 교육의원.

level (구분 컬럼 기반): 당일투표, 사전투표(관내), 관외사전투표, 거소선상. 집계행(합계/소계)은 저장하지 않고 검증의 기준값으로만 사용.

## 빌드 순서

검증 모듈을 먼저 만들고(파서들의 채점 기준), 회차별 파서를 하나씩 통과시킨 뒤 다음으로.

1. `validate_local.py` — 검증 게이트
2. `parse_8th` → 검증 통과 (통합파일 + 헤더갱신, 지금 깨진 것)
3. `parse_7th` → 검증 통과 (통합파일, 8회와 유사)
4. `parse_6th` → 검증 통과 (시도별 분리)
5. `parse_5th` → 검증 통과
6. `parse_4th` → 검증 통과
7. `parse_3rd` → 검증 통과 (XLS, A형/B형)

## 서브에이전트 운용

각 파서는 Opus 서브에이전트가 담당. 각 에이전트는:
1. 자기 회차 원본을 직접 열어 행 구조·헤더 위치·시도 전환 지점 파악
2. 파서 작성
3. `validate_local.py`로 자가 검증 — 통과할 때까지 반복
4. 검증 리포트(초록불)와 함께 복귀

검증 모듈이 먼저 존재해야 서브에이전트가 스스로 채점하며 고칠 수 있다 → 1번이 선행.

## 성공 기준 (각 파서 공통)

- `validate_local.py` 전 항목 통과 (시도별 후보 상이 · 합계 정합 · 중복 없음 · 범위 정합)
- 시도지사는 선관위 공식 시도별 1위 득표와 대조 통과 (가능한 회차)

## 기존 코드 처리

현재 `process_local_elections.py`(63KB, 9샘플 검증)는 참고용으로 남기되, 새 파이프라인은 깨끗한 모듈로 새로 작성한다. 기존 파일을 직접 수술하지 않는다 — 어디가 망가졌는지 추적 불가능한 상태라 새로 짜는 게 빠르다.

`analyze_twin_votes.py`(쌍둥이 분석)와 웹 UI는 **검증된 CSV가 나온 뒤** 그 위에서 재실행한다. 이번 spec 범위는 검증된 CSV 생성까지.
